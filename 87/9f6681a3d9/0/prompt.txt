## Technical Context

### Architecture:
- **rmcp proc-macro MCP pattern**: `#[tool_router]` on `impl McpServer` auto-generates tool routing. New tools added via `#[tool]` methods. Each takes `Parameters(args): Parameters<SomeArgs>` where `SomeArgs: Deserialize + JsonSchema`. `#[tool_handler]` on `impl ServerHandler for McpServer` wires everything.
- **Ouros sandbox initialization timing**: External function names must be registered at `SessionManager::new()` / `create_session()` time via `mgr.reset(None, funcs.clone())?` and `mgr.create_session(MAIN_SESSION_ID, funcs)?`. Custom tool names are fed in at this point via `all_external_functions()`, which merges builtins + registry names with dedup.
- **Interior mutability pattern**: `McpServer` derives `Clone` and holds `tool_registry: Arc<Mutex<ToolRegistry>>`. The `configure_tools` MCP tool mutates the shared registry; `run_chat_rlm_query` clones it into `RlmConfig` before the RLM run. This ensures the registry snapshot is consistent for the duration of a query.
- **Custom tool dispatch**: `handle_external()` in `rlm.rs` has a large `match` block for builtins. The fallback arm (`other =>`) now checks `self.config.tool_registry.get(other)`. If found, returns a passthrough message `"Tool '{}' called with args: [...] — no handler registered on server side"`. If not found, returns the original `"Error: unknown function '{}'"`.

### Key decisions:
- **Passthrough handler model for custom tools**: Currently, custom tool calls in the sandbox return an informational string rather than executing a real handler. This is the initial "registration + prompt injection" phase. Actual handler execution (e.g., MCP callback to the caller) would be a future extension.
- **Registry is cloned into RlmConfig per query**: This avoids lifetime/shared-reference issues and ensures the sandbox sees a consistent tool list even if `configure_tools` is called concurrently.
- **Custom tool names are deduped against builtins**: `all_external_functions()` skips adding a name if it already exists in the builtin list, so registering a tool named `"FINAL"` won't create a duplicate entry in the external functions list.

You’d make Axon the agent’s “model provider,” not just a tool.

The clean shape is:


agent
  -> OpenAI-compatible /v1/chat/completions or /v1/responses
      -> Axon HTTP adapter
          -> ContextStore thread
          -> Rlm::completion(query, context)
              -> recursive llm_query(...)
              -> sandbox/tool execution
          -> OpenAI-shaped response
  <- assistant message
In this repo, the core call already exists: 
rlm.rs
 exposes Rlm::completion(query, context). The MCP server in 
mcp.rs
 already does most of the orchestration: read a thread from ContextStore, build RlmConfig, run the RLM, append USER/ASSISTANT back into memory.

So I would add a new HTTP mode beside MCP, probably serve-openai, with routes like:


POST /v1/chat/completions
POST /v1/responses
GET  /v1/models
The first version should support the smallest useful OpenAI subset:


{
  "model": "axon-rlm",
  "messages": [
    {"role": "system", "content": "..."},
    {"role": "user", "content": "..."}
  ],
  "stream": false,
  "metadata": {
    "thread_id": "repo-axon"
  }
}
Map it internally like this:


let thread_id = req.metadata.thread_id.unwrap_or(hash_of_messages_or_default());
let stored_context = store.read_context(&thread_id);

let query = extract_latest_user_message(&req.messages);
let request_context = render_prior_messages_and_system(req.messages);
let context = format!("{stored_context}\n\n[CURRENT REQUEST]\n{request_context}");

let answer = rlm.completion(&query, &context).await?;

store.append_context(
    &thread_id,
    &format!("\nUSER: {}\nASSISTANT: {}\n", query, answer),
)?;
Return something shaped like Chat Completions:


{
  "id": "chatcmpl-...",
  "object": "chat.completion",
  "created": 1777130000,
  "model": "axon-rlm",
  "choices": [
    {
      "index": 0,
      "message": {
        "role": "assistant",
        "content": "..."
      },
      "finish_reason": "stop"
    }
  ]
}
For responses, do the same thing but wrap the answer as output text. I would not try to perfectly implement every OpenAI Responses feature at first. Agents usually need basic text compatibility, model listing, errors, and maybe streaming.

The key design choice: make thread_id explicit. OpenAI requests are often stateless, but RLM wants durable context. Put thread_id in one or more places:


metadata.thread_id
user field
header: X-Axon-Thread
fallback: default
For an agent, this is the big win. Every model call becomes communication with the shared RLM memory, not an isolated prompt.

Streaming is tricky. Axon currently returns only the final String from Rlm::completion. You can fake OpenAI SSE by waiting for the full answer, then streaming chunks, but that only helps client compatibility. True useful streaming would require changing 
rlm.rs
 to emit events:


rlm.started
llm.iteration_started
sandbox.code_executed
rlm.child_started
rlm.child_finished
message.delta
message.completed
For agents, I’d start non-streaming. RLM calls are recursive and higher-latency anyway; pretending it is a normal token-streaming model may hide what makes it valuable.

Tool calls need a hard decision. Normal OpenAI agent loops expect:


model requests tool call
agent executes tool
agent sends tool result back
model continues
Axon’s model is different:


RLM emits repl code
sandbox executes code
sandbox may call llm_query or registered tools
RLM continues internally
So for “RLM as primary communicator,” I’d let Axon own tool use internally. The OpenAI-compatible endpoint should mostly return final assistant text. Later, you can translate OpenAI tools from the request into Axon’s ToolRegistry, but the agent must understand that tool execution may happen inside Axon rather than in the outer client loop.

Implementation-wise, I’d factor the duplicated MCP logic into an RlmService:


pub struct RlmService {
    client: Client,
    model: String,
    sub_model: String,
    max_iterations: usize,
    max_depth: usize,
    store: ContextStore,
    policy_catalog: PolicyCatalog,
    default_runtime_policy: RuntimePolicy,
    tool_registry: SharedToolRegistry,
}

impl RlmService {
    pub async fn query(&self, req: RlmQueryRequest) -> Result<RlmQueryResponse> {
        // read context, build policy, create Rlm, call completion, append transcript
    }
}
Then MCP and OpenAI HTTP both call the same service. That avoids two subtly different Axon runtimes.

The minimum viable path:

Add src/openai.rs with request/response structs and Axum routes.
Add Commands::ServeOpenAi { bind } in 
main.rs
.
Extract the shared “read context -> run RLM -> append context” logic out of 
mcp.rs
.
Implement /v1/chat/completions first.
Add /v1/models.
Add fake SSE only if the target agent refuses non-streaming responses.
Add /v1/responses after chat completions works.
The core idea: don’t make the RLM “another tool” the agent sometimes calls. Make it the base URL the agent talks to. The agent thinks it is calling a model; Axon turns every model call into a persistent recursive reasoning session.
